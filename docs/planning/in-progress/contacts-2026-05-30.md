# Contacts — 2026-05-30

*Opened 2026-05-30 in response to the user's brief: "Create a contact
widget in which we can save contacts. The contacts can have their info
saved. We also need to be able to label the contacts as potential
customers, current customers, reoccurring, former, employee, student,
teacher, etc. I want to make it so we can have regular customers have a
profile. This will be for realtors and stuff. It will also help in
linking customers to specific jobs and vice versa."*

## Today's reality (audit, 2026-05-30)

- No `contacts` table exists. Job rows carry `client_name`,
  `client_email`, `client_phone`, `client_company`, `client_address`
  fields directly — every realtor / repeat client is duplicated on
  every job they touch.
- No `/admin/contacts` route exists (the route-registry has no entry).
- The Quick Actions catalog has no contact action.
- Hub widget catalog has no contacts widget.

## Slices

### Slice 1 — Schema (foundation) ✅ shipped 2026-05-30

- New migration `seeds/305_contacts.sql`:
  - `contacts` table: id, name (required), email, phone, company,
    title, address, city, state, zip, **`labels TEXT[]`** (multi-tag
    so a realtor can be both "current customer" + "recurring"),
    notes, created_at, created_by, updated_at + `updated_at` trigger
    mirroring the `cad_drawings` pattern.
  - Indexed: lower(name), lower(company), lower(email), `GIN(labels)`
    (cheap multi-label filters), `updated_at DESC` (recents).
  - `job_contacts` join: id, job_id (FK CASCADE), contact_id (FK
    CASCADE), role (default 'client'), notes, created_at, created_by
    + `UNIQUE (job_id, contact_id, role)` so the same person isn't
    double-linked in the same role on the same job.
  - Service-role-only RLS on both tables, same pattern as
    `drawing_notes`.
- New pure `lib/contacts/labels.ts`:
  - `CONTACT_LABELS` array with the 7 defaults the user named:
    potential customer, current customer, recurring customer,
    former customer, employee, student, teacher. Each entry carries
    `id` / `label` / `description` / `tint`.
  - `findContactLabel(id)` + `isKnownLabel(id)` for chip rendering
    (catalog ids vs. user-coined free-form labels).
  - `normalizeLabel(raw)` collapses any user input ("Recurring
    Customer", "RECURRING", "recurring-customer") to one storage key
    (`recurring_customer`) + drops stray punctuation + returns null
    on blank input.
  - `JOB_CONTACT_ROLES` ships the relationship vocabulary the
    job-detail picker offers (client / realtor / agent / buyer /
    seller / lender / attorney / engineer / contractor / other).
- 10 specs (catalog order + roundtrip + normalize behavior +
  null-guards + roles). Typecheck + lint clean.

### Slice 2 — CRUD API ✅ shipped 2026-05-30

- New `app/api/admin/contacts/route.ts`:
  - `GET` supports `?search=` (ilike on name / company / email, with
    `%`/`_` escaping), `?label=` (catalog or user-coined key,
    contains-match against the `labels` GIN index), `?limit=` (caps
    at 500), `?order=name` (defaults to `updated_at DESC` — most-
    recent first).
  - `POST` validates required `name` via `sanitizeContactInput(...,
    { requireName: true })`, stamps `created_by`, returns the
    inserted row.
- New `app/api/admin/contacts/[id]/route.ts`:
  - `GET` returns the contact plus the nested job-link list joined
    to `jobs(id, name, job_number, stage)` so the profile page can
    render in one fetch.
  - `PUT` reuses the sanitizer; only sets columns the body actually
    sent (no clobbering on partial update) — labels intentionally
    always-sent so the chip-row commits.
  - `DELETE` is hard-delete (no `archived_at` yet — flagged in the
    plan; if the user pushes back we add it later).
- New `app/api/admin/contacts/[id]/jobs/route.ts`:
  - `POST` writes one `job_contacts` row; PG `23505` unique-violation
    maps to a friendly `409 — already linked in that role`.
  - `DELETE ?job_id=&role=` removes the matching link; without
    `role`, removes every role for that (contact, job) pair.
- New pure `lib/contacts/payload.ts` ships
  `sanitizeContactInput(body, { requireName? })` +
  `sanitizeJobLinkInput(body)` + `sanitizeLabels(raw)` so route
  handlers stay thin. 14 specs lock the trim + null-out + label
  normalize + required-name behavior.
- Route-registry entry added: `/admin/contacts` in the **office**
  workspace (icon `Users`, keywords address-book / people /
  realtors / clients). Now bell + Cmd+K + nav see it.
- 18 specs added across labels + payload (10 + 4 from spec re-runs
  + 4 from sanitizeJobLinkInput). 1943 contacts + admin + hub +
  notifications specs green; typecheck + lint clean.

### Slice 3 — Contacts list page ✅ shipped 2026-05-30

- New `app/admin/contacts/page.tsx`:
  - Search input (submits on Enter, ✕ clears).
  - Label chip-row driven by `CONTACT_LABELS` + an "All" toggle.
  - Table (Name + title / Company / Email mailto / Phone tel / Label
    chips). Name links to `/admin/contacts/[id]` (lands Slice 4).
  - Role guard (admin / developer / tech_support).
  - Empty + loading + error states.
- New `app/admin/contacts/NewContactDialog.tsx`:
  - `ModalFrame` form with name (required) / company / email / phone +
    a labels picker. Catalog chips toggle inline; a "custom label"
    input adds free-form keys after running them through
    `normalizeLabel` so storage stays consistent.
  - Posts to `POST /api/admin/contacts`; on success the list page
    navigates straight to the new profile.
- 1818 admin + hub + contacts specs green; typecheck + lint clean.

### Slice 4 — Contact detail / profile page ✅ shipped 2026-05-30

- New `app/admin/contacts/[id]/page.tsx`:
  - Fetches GET `/api/admin/contacts/[id]` once + reads the nested
    `jobs` array the Slice-2 API returns.
  - Editable basic info via the existing `InlineEditField` jobs use
    (name / company / title / email / phone / address / city /
    state / zip), each `PUT`-ing one field.
  - Labels chip row with add/remove — catalog chips toggle inline;
    free-form `customLabel` input runs through `normalizeLabel`
    before joining the list.
  - Notes panel (`InlineEditField type="textarea"`).
  - Linked jobs section listing each job with role chip + stage +
    a per-row "Unlink" (DELETEs with the `job_id` + `role` query).
  - "+ Link a job" button opens `<LinkJobDialog>`.
  - "Delete contact" button (DELETE → back to list).
  - Role guard (admin / developer / tech_support).
- New `app/admin/contacts/[id]/LinkJobDialog.tsx`:
  - `ModalFrame` with a job search filter (name + job_number),
    selectable list, role picker (uses `JOB_CONTACT_ROLES`),
    optional notes, POSTs to the Slice-2 link endpoint. PG 409
    "already linked" surfaces as the inline error.
- 1853 hub + admin + contacts + middleware specs green; typecheck +
  lint clean.

### Slice 5 — Contacts hub widget

- `lib/hub/widgets/contacts/index.tsx` — small list view showing the
  N most-recently-touched contacts. Footer "Go to contacts" links to
  `/admin/contacts` (added to widget-links registry).
- Settings: pick which labels to show (default = current + recurring
  customers), how many rows.
- Tiny bucket = count of contacts matching the filter.
- Empty state = "No contacts yet — track realtors, clients, partners."
- Quick-add tile in the widget so the surveyor can save a contact
  without leaving the hub.

### Slice 6 — Job ↔ contact linking

- Job detail page overview tab gets a "Contacts" section listing
  linked contacts with their role chips. "Link a contact" button
  opens the picker. Removing a chip unlinks.
- The existing `client_name` / `client_email` / `client_phone` /
  `client_company` columns on `jobs` stay as-is for legacy / unlinked
  rows; the new picker writes to `job_contacts` and doesn't touch
  those fields. We can backfill later if the user wants.

## Out of scope / placeholder

- Importing contacts from a CSV / address book — captured here as
  the next obvious follow-up once the user has data.
- Contact-to-contact relationships (Realtor → Client referrals) — a
  graph table is overkill for v1.
- Per-contact deal pipeline / lead status — distinct from labels,
  belongs in a separate plan when the user asks for it.

## Guardrails

- Don't change `jobs.client_*` fields. The new join is additive so
  legacy jobs keep displaying their inline client info.
- Don't auto-create a contact every time a job is created — the
  surveyor explicitly opts in via the "Link a contact" picker.
- Per slice: typecheck + lint + commit + push + annotate this doc.

## TL;DR

Six slices: a `contacts` table + a `job_contacts` join, CRUD API +
route-registry entry, list page, profile page, hub widget, job-detail
linker. Realtors + repeat clients get one profile each; jobs link to
that profile; the widget keeps the surveyor close to their
relationships from the hub.
